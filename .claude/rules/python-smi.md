# Python / SMI ルール

Python 側は主に **SMI 拡張マッチングソルバ**と、必要に応じた AI オーケストレーション補助に使う。技術アピール §1.1 の根幹のため、**品質基準を高く**保つ。

## スコープ

- `algorithm/` — SMI 本体、ユーティリティ、ベンチマーク
- `api/` — マッチング API (Next.js Route Handler から呼び出すか、Supabase Edge Functions 経由)
- `tests/` — ユニットテスト / プロパティベーステスト

## 言語 / ツール

- **Python 3.11 以上**
- フォーマッタ: `ruff format`
- Linter: `ruff check` (strict)
- 型: `mypy --strict` で通る。`Any` を使わない
- テスト: `pytest` + `hypothesis` (プロパティベース)

## SMI 実装方針

1. **外部マッチングライブラリ禁止**。`matching`, `networkx` のマッチング機能を使わない。自前実装が技術アピールの根拠。
2. **男女比均等化はハード制約**として実装。SMI の比較関数に制約違反ペナルティを入れず、枠割り当てフェーズで物理的に弾く。
3. **Min-Regret Sum** を実装する。全参加者の後悔値総和を最小化する変種。通常の SMI と両方のバリアントを切り替え可能にする。
4. **不完全リスト (Incomplete Lists)** を扱う: 「誰にも投票しない」「3 人に投票」どれも壊れない。
5. **男女比動的リバランシング**: 参加者数が変動したときにハード制約を再計算するインクリメンタル更新を提供する。

## テストの必須ケース

- SMI 定理: 出力がペア内で stable である (ブロッキングペアが存在しない)
- Min-Regret Sum: 固定入力に対して既知の最適解を返す (ゴールデンテスト 3 ケース以上)
- 男女比境界値: 3 vs 3、3 vs 5 (5 対 3 の逆も)、0 vs N
- 非二元 / 未回答ユーザ: `non_binary` が別プールで処理される / `prefer_not_to_say` は均等化制約から除外される
- 投票 0 人ユーザ: SMI が破綻せずに完了する
- プロパティ: 「同じ入力で 2 回実行すると同じ出力」(決定性)

## コード品質

- 関数は 50 行以内、ファイルは 400 行以内目安
- 早期 return。ネスト 3 段まで
- Dataclass or Pydantic モデルで入出力を型定義する。生 dict を外部境界から内部に伝播させない
- 副作用のある関数と pure function を明確に分離する (純粋関数側に SMI 本体を書く)

## パフォーマンス

- N=200 (イベント最大) で 1 秒以内に完了すること
- ベンチマークは `pytest --benchmark` で記録し、回帰したら CI で落とす
- 可視化向けに**各反復のスナップショット**を列挙できる iterator 版を用意する (フロント側のノード/エッジアニメーション用)

## 禁止事項

- `print` によるデバッグをコミットしない (`logging` を使う)
- Mutable なグローバル状態を持たない
- SMI 本体に I/O を混ぜない。I/O は呼び出し側のアダプタレイヤ
