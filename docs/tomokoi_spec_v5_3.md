## 0. マニフェスト

## ── 透明にされた者たちへ ──

この世にはルッキズム・スペック至上主義の象徴たるマッチングアプリが蔓延っている。

OkCupidが自社データを解析したとき、一つの冷たい事実が浮かび上がった──**女性は男性の80%以上を「平均以下」と評価していた**。逆方向ではそんな偏りは起きていない[1]。Hingeの内部データを解析したGoldgeier(2017)でも、マッチは上位の男たちに**約10倍集中**している。そしてTinderは月額で「Boost(表示10倍)」を売り、Hingeは「Boost(11倍)」を売り、Bumbleは「Spotlight」を売る──各社公式ヘルプに、堂々と。つまりプラットフォーム自身が、**「上位だけ可視化、下位は透明化」を商品として売りさばいている**わけだ。

しかし、**マチアプだけが悪いのではない**。マチアプは症状であって、病そのものじゃない。現代社会の腐った一面を反映しているに過ぎない。

---

**原因はSNSだ。**

世界中の他人と一瞬で比較できるようになり、フォロワー数という一次元の数字が、人間の価値を測るステータスに収束した。そしてそのランキング上位を、ルッキズム・スペック至上主義の勝者たちが占拠した。我々は朝から晩まで、彼らのタイムラインをスクロールしている。そうして我々の価値観が徐々にルッキズム・スペック至上主義に塗り替えられていく。

これが、社会の腐った一面の正体だ。

---

**放置すれば、社会は二つに裂ける。**

片方は、ルッキズム・スペック基準で選ばれなかった下位層。そもそも土俵に上がれない。出会いそのものを失い、結婚機会を奪われていく。

日本では**18〜34歳の未婚男性の72.2%が、異性の交際相手を一人も持っていない**[2]。10人のうち7人だ。
アメリカでは18〜29歳で「過去1年間に性交していない」割合は、**12%(2010)から24%(2024)へ倍増**した[3]。

ピラミッド構造を考えれば、下位層は往々にして多数派だ。だから、少子化は静かに、しかし確実に加速する。

もう片方は、選ばれた上位層だけによるクローズドな結婚市場。容姿・スペックの高さが日常の標準になった家庭で、子が育つ。45カ国N=14,399の事前登録追試(Walter et al., 2020, _Psychological Science_)が示したように、「経済力・スペック重視」という配偶者選好の性差は、平均傾向として確かに存在する。マチアプはそれを過剰に可視化し、増幅する。親の基準は、**そのまま子どもに受け継がれる**。そしてその子どもたちがSNSの次の上位層となり、次世代のルッキズム・スペック至上主義の担い手になる。

ループは閉じる。

---

**こんな優しさの欠けた世界を、僕は否定する。**

この連鎖を、どこかで断ち切らなければならない。何百万人分の「透明にされた者たちの人生」を、そのまま次世代に引き継がせるわけにはいかない。

だからこそ**トモコイ**が必要だ。

容姿もスペックも、最初は公開しない。
友達からの紹介という、身近で信頼できる入り口から。
性格・趣味・価値観・波長で、人と人をつなぐ。

ジャッジの基準を、スワイプ0.3秒の顔写真から、**その人を何年も知っている友達の推薦**へと、ひっくり返す。

下位8割の男性はマッチングしない、女性は食い散らかされて終わる──そんなマチアプ敗北者を、僕たちはごっそり奪う。腐った社会を助長して私腹を肥やしてきた老害どもには、ご退場願おう。

## 1. プロダクトビジョン

### 1.1 一言説明

> 「友達があなたをアバターでプレゼンしてくれる、ライブイベント型オンラインマッチング」

### 1.2 コアコンセプト

海外で急拡大中の **Date My Mate**（友人がスライドで独身の友人を紹介するライブイベント）を日本初上陸させる。単なる移植ではなく、**参加者全員がVRM 3Dアバター**で登場することで、容姿情報を扱わずに性格・趣味・価値観で繋がる体験に再設計する。さらに **紹介者メイン + 被紹介者リアクション** の掛け合い型ピッチを軸に据え、VTuberコラボ配信的な盛り上がりを意図的に作り込む。

コンセプトを一行で言えば: **「友達があなたをアバターでプレゼン、ライブイベント型マッチング」**。

### 1.3 解決する課題

**マクロ課題**:

- 出会いの1位だった「友人紹介」が30.8%（2015年）→ 10.9%（2024年）に急落
- マッチングアプリ（32.2%）と完全逆転、しかし利用者の9割が「アプリ疲れ」
- 月平均18時間の「恋活ロストタイム」
- 少子化の根本原因としての出会いの減少

**ミクロ課題（既存マチアプの構造的欠陥）**:

- スペック比較疲れ
- スワイプの虚無感
- 一瞬の外見判断による偏り
- 自己PRの苦痛（日本人の謙遜文化と不一致）

### 1.4 提供価値

- **反ルッキズム**: アバター化により容姿・スペック情報を完全遮断
- **信頼**: 友人による第三者紹介で信憑性を担保（なりすまし流入も構造的に弾ける）
- **謙遜文化への適合**: 自己PR不要、友人が代わりに語る
- **エンタメ性**: イベント型の「場の力」、VTuberコラボ配信的な**掛け合い**で出会いを創出
- **お節介文化の復活**: 消滅した友人紹介文化をテクノロジーで再構築

### 1.5 ハッカソンテーマ「小さくなる日本」との接続

少子化の根本原因＝出会いの減少。マチアプがルッキズム競争を加速させた結果、8割の敗者と、容姿・スペック至上主義の子供世代を生む。トモコイはこのスパイラルを断ち切る。

## 5. ユーザー体験フロー

### Phase 1: 事前準備

1. アカウント登録（メール・ニックネーム・年齢・性別・趣味・居住地）
2. **VRMアバター選択**（プリセット10種から、カスタマイズなし）
3. イベント閲覧 → エントリー
4. プレゼン枠は友人ペアで登録

### Phase 2: スライド作成（AI生成ベースの単一フロー）

紹介者が被紹介者を紹介するスライドを作成する。**スライドは掛け合い誘発型に設計されている**ため、本番のピッチでは紹介者と被紹介者の2人で進行することを前提に作る。

**単一フロー**:

1. **テンプレート選択**（掛け合い誘発型テンプレ3〜5種類から、スライド5枚構成）
2. **質問に答える**（紹介者が被紹介者について回答。各スライドに対応した質問セットあり。被紹介者への質問はしない）
3. **AI生成**（Gemini 3 Flash がテンプレに沿って掛け合い誘発型の文面を生成）
4. **編集**（生成結果を紹介者がフィールド単位のエディタUIで編集。画像スロットに任意画像を貼り付け可能）
5. **被紹介者への確認依頼**（承認・修正依頼・辞退）
6. **主催者による事前チェック**（公開前承認制、NG内容があれば差し戻し）
7. **確定・公開**（ストレージに本番用として保存、以降イベントで表示）

**AI非採用から採用への方針転換（v5.2）**: v4.1〜v5.0 では「友人が自分の言葉で友達を語る」という Move Hands の思想からAI生成を不採用としていた。v5.2 では **AIを整形役・語り手は紹介者** と位置づけ直す。質問には紹介者が自分の言葉で答えることを必須とし、AIは回答をテンプレに沿った掛け合い誘発型スライドに整形するだけ。語り手は最後まで紹介者のまま。

**お土産エクスポート（任意）**: 完成スライドは紹介者が**PPTX形式でダウンロード可能**。これは保存・共有・記念用の一方通行エクスポートであり、編集して再アップロードする機能はMVPでは提供しない（Phase 2送り）。本番表示はWebレンダリング一本。

## 6. 重要な設計判断

### 6.1 実写写真・個人情報の非取得

プラットフォーム内ではマッチングするまで実写写真・本名・連絡先を要求しない。反ルッキズムを最後まで貫徹し、データ最小化（プライバシーバイデザイン）も同時に達成する。

### 6.2 アバター方針（VRM 3D）

**採用技術**: VRM フォーマット + MediaPipe表情トラッキング

**選定理由**:

- 既存の無料VRMアバター（VRoid Hub等）が豊富
- チームに恋AIでの3Dアバター + MediaPipe経験あり（技術的非対称優位）
- POPOPOの2026年リリースで「アバター通話」文化が日本で拡大中
- 審査員へのビジュアルインパクトが圧倒的

**運用方針**:

- プリセット10種から選択、カスタマイズ不可
- ユーザー自作VRM持ち込みは Phase 2以降
- ライセンス商用可のものを事前厳選

### 6.3 テーブル人数（3〜4人、男女バランス重視）

**制約付きk-partition 2-opt自前実装**により、被紹介者を1テーブル 3〜4人に配置する。**紹介者は交流タイムの卓に参加しない**（ラウンジ/休憩エリアへ）。イベント参加は紹介者+被紹介者の2人組必須。

- マッチング対象は被紹介者のみ（紹介者不参加により登壇ペア同卓は自動充足）
- テーブルサイズ {3, 4}、被紹介者 N=5 のみ {5} を例外許可
- **男女 2:2 はソフト最優先（辞書式①）**、ハード制約ではない
- 同性紹介ペアも許容（LGBTQ+対応）
- 3〜4人は会話が成立する最小構成、各人が2〜3名と会話可能
- 制約付き最適化問題として数学的に美しく定式化できる
- 4人同時アバター表示はWebGL負荷の許容範囲
- **全被紹介者がいずれかのテーブルに必ず配属される**

### 6.4 紹介者の「推薦フラグ」

「紹介者がマッチング相手を選ぶ」案は**不採用**（被紹介者の自己決定権を尊重）。

**代替案**: 推薦フラグ

- 被紹介者のマッチング画面に星マークで推薦表示
- 紹介者は最大3名まで推薦可能
- アルゴリズム入力には含めず、参考情報に留める

お節介文化の復活であり、お見合い強制ではない。

### 6.5 プレゼン形式：掛け合い型ライブピッチ

**掛け合い型ライブピッチ一択**（ひとこと版・録画版は不採用）。紹介者がメインで語り、被紹介者が **合いの手・ツッコミ・照れリアクション・友達クイズの回答・本人からひとこと** を担当する掛け合い構成。

**設計根拠**:

- Date My Mate海外版のバズ動画の核は **2人の掛け合い**にある。被紹介者が音声OFFで照れ顔だけの構成では、この核が取り逃される
- VTuberコラボ配信的な盛り上がりを意図的に設計する
- スタンプが最も飛ぶのは掛け合いが決まった瞬間 → 明確な盛り上がり設計になる
- 日本の謙遜文化への配慮は「自己PR不要」（自分で自分を語らない）の部分で果たされており、被紹介者が語らない必要はない

### 6.6 アワード（スタンプ連動）

- 「盛り上がったで賞」: **プレゼン中のスタンプ獲得総数最多の紹介ペアに自動表彰**（MVP）
- **「異性から人気1位」系は絶対に設けない**（空気を悪くするため）
- 「おもしろいで賞」「友情が深まったで賞」は Phase 2 以降で実装予定

### 6.7 安全性 → §17 参照

**→ §17「信頼・安全設計」で段階的施策として詳述。**

本章では方針の要点のみ:

- 顔出しなしマッチングには「おじさん凸」「詐欺師流入」のリスクが構造的に存在する
- **紹介者の実名性 × 被紹介者の匿名性** というアシンメトリを実装レベルで担保する
- マイナンバーは思想的整合性の観点から不採用（「容姿を隠す」と「国民番号で身元確認」は相性が悪い）
- 大学・企業メールドメイン認証を主軸に据える（ハードルはマイナンバーの1/10、なりすましコストは十分に高く、初期ターゲット層とも完全一致）

### 6.8 スライド作成のAI生成採用と単一フロー設計

紹介者がスライドを作る方法を、**AI生成ベースの単一フロー**に集約する。v4.x では AI を完全不採用、v5.1 では AI生成/アップロードの2ルート設計としていたが、v5.2 で設計を整理する。

**方針転換の経緯**:

- **v4.x（AI完全不採用）**: AI生成 = 「友人の言葉」を機械に置き換える → 思想と矛盾 → 不採用
- **v5.1（2ルート構成）**: AIを整形役と位置づけ直したうえで、AI生成ルートとアップロードルートを併置
- **v5.2（単一フロー + お土産エクスポート）**: 2ルート併置は過剰設計と判断し、AI生成ベースの単一フローに一本化。編集自由度は**フィールド単位のエディタUI**と**任意画像スロット**で担保し、完成物は**PPTX形式でダウンロード可能**にして保存・共有の自由度も確保

**単一フロー化の論理**:

- アップロードルートとAI生成ルートは、実装の大半（テンプレ構造・編集UI・被紹介者確認・主催者チェック・本番レンダリング）が共通
- ルート分岐は UX の複雑化と開発工数の増加を招くだけで、ユーザーの実質的な選択肢を増やさない
- 「凝ったスライドを作りたい」ニーズは、**完成後のPPTXダウンロード**で別の形に満たせる（個人的な保存・編集用途として）
- 本番レンダリングは Web の React コンポーネント一本で、Supabase Realtime による同期表示との整合を崩さない

**AI生成の思想的整合性**:

- 質問には**紹介者が自分の言葉で答える**ことを必須（空欄でのAI丸投げはUI上不可）
- AIは回答を掛け合い誘発型テンプレに落とし込む整形役に徹する
- 語り手は最後まで紹介者のまま → Move Hands の本質を保持

**編集自由度の担保**:

- フィールド単位で文面を再編集可能（紹介者セリフ・被紹介者リアクション想定の両方）
- スライド単位でのAI再生成ボタン（同じ質問回答から別の文面案を得る）
- 画像スロットに任意画像を貼り付け可能（友達の好きなものの画像など。AIは画像を見ない設計）
- PPTXダウンロードで、紹介者は本番後にPowerPoint上で自由に編集して記念保存できる

**ガードレール**:

- NGカテゴリ（容姿・スペック）は、AIシステムプロンプトでの抑制 + 主催者の人力チェックで二重に防御
- 被紹介者は本番前に必ず完成スライドを確認・承認する（拒否権あり）
- 主催者による公開前承認が最終ゲート

詳細は §7.3 参照。

### 7.3 スライド作成（AI生成ベースの単一フロー、掛け合い誘発型）

紹介者はAI生成ベースの単一フローでスライドを作成する。完成後、被紹介者確認 → 主催者チェック → 確定・公開の順に進む。完成スライドはPPTX形式でダウンロード可能（お土産エクスポート）。

**テンプレート構成（全5枚、掛け合い誘発型 + 内面エピソード誘導型）**:

| #   | テーマ                          | 内面エピソード誘導の方向性                     | 紹介者パート                                               | 被紹介者パート                                           | 画像スロット       |
| --- | ------------------------------- | ---------------------------------------------- | ---------------------------------------------------------- | -------------------------------------------------------- | ------------------ |
| 1   | 一言で言うと、こんな人          | その人といる時の「気持ち・空気感」を言語化     | 「○○さんは、一緒にいると \_\_\_ な気持ちになる人です」     | 軽い照れリアクション・ひとこと（「言いすぎだって〜」等） | 0枚                |
| 2   | 好きなこと・ハマってること      | **熱量のある具体エピソード**を引き出す         | 「最近 **_ にハマっていて、この前なんて _**」              | 補足・エピソード追加（「ちなみに昨日も〜」）             | 1枚（任意）        |
| 3   | **意外な一面・ギャップ**        | 第一印象と違う面、人に見せない素の姿           | 「普段は **_ なのに、実は _** なところがあって」           | ツッコミ or 肯定（「バレたか〜」等）                     | 0枚                |
| 4   | 友達クイズ                      | その人ならではの価値観・選択基準を可視化       | 「この人が必ず迷う二択は？ A: **_ vs B: _**」              | 回答発表 + 理由の掛け合い                                | 2枚（任意、A/B用） |
| 5   | 紹介者の締め → 本人からひとこと | **心に残ったエピソード**で締め、本人の声で返す | 「私が \_\_\_ のエピソードで、この人のこと大切だと思った」 | **本人からひとこと**（意気込み・挨拶）                   | 1枚（任意）        |

**v5.3 での強化点**: Slide 3 を「合いそうな人」から **「意外な一面・ギャップ」** に変更。これは内面マッチングの核となるスライドで、表面的なプロフィールでは絶対に伝わらない情報を引き出す設計。Slide 2 と Slide 5 も質問を**エピソード具体化の方向**に振り直した（「ハマっている」→「どんなエピソードが」、「大切な友達」→「どんな瞬間に大切だと感じた」）。

各スライドに **「紹介者セリフ案」「被紹介者のリアクション想定」** の2枠を用意し、ペアで事前に読み合わせができるようにする。テンプレは上記の5枚構成を基本とし、**複数バリエーション**（真面目トーン / カジュアルトーン / クイズ強化型 等、MVPで3〜5種類）を用意する。

**内面エピソード誘導UIの工夫（§7.3.5 で詳述）**:

- 質問フォームに「容姿・スペックではなく、具体エピソードを書いてください」というガイド
- 回答が短すぎる場合は「もう少し詳しく教えてください」リアルタイムヒント
- 「良い回答例」「避けてほしい回答例」をUI上に併記

#### 7.3.1 作成フロー

```
1. テンプレート選択（3〜5種類から、スライド5枚構成）
         ↓
2. 質問に答える（紹介者のみ回答）
   各スライドに紐づく質問セットに紹介者が入力
   例: Slide 1 → 「その友達と一緒にいるとどんな気持ちになる？」
       Slide 2 → 「その友達が最近ハマっていることを教えて」
       Slide 4 → 「その友達が必ず迷う二択があれば」
   被紹介者への質問はしない（負担削減 + 「友人が語る」構造の維持）
   必須入力項目あり（AIへの丸投げ防止）
         ↓
3. AI生成
   Gemini 3 Flash が質問回答をテンプレに沿った
   掛け合い誘発型スライド文面に整形
   紹介者セリフ + 被紹介者リアクション想定 の両方を生成
   構造化JSON出力で返却（responseSchema 指定）
         ↓
4. 編集（フィールド単位のエディタUI）
   生成結果をUI上でフィールド単位に編集
   - 紹介者セリフ・被紹介者リアクション想定のテキスト編集
   - スライド単位・フィールド単位のAI再生成ボタン
   - 画像スロットへの任意画像アップロード（Slide 2/4/5）
     ※ AIは画像を見ない（紹介者が選んで配置するだけ）
         ↓
5. 被紹介者への確認依頼（§7.3.3）
         ↓
6. 主催者による事前チェック（§7.3.3）
         ↓
7. 確定・公開 → ストレージに本番用として保存
         ↓
[任意] PPTXダウンロード（§7.3.2）
```

#### 7.3.2 お土産エクスポート（PPTXダウンロード）

完成後、紹介者は**スライドをPPTX形式でダウンロード可能**。

**位置づけ**:

- 保存・共有・記念用の**一方通行エクスポート**
- 本番イベントでの表示には使われない（本番はWebレンダリング一本）
- 編集して再アップロードする機能はMVPでは非提供（Phase 2送り）

**実装**:

- `pptxgenjs` ライブラリで Server Action 経由生成
- Node.js ネイティブのため Vercel Server Action から直接呼び出し可能
- テンプレに応じた基本レイアウト（背景・フォント・画像配置）を埋め込み
- 日本語フォント対応（Noto Sans JP などを埋め込み）

**MVP優先度**: Week 3 後半のNice-to-have扱い。Week 3で基本機能が安定したら実装。時間切れの場合は本番機能を優先し、PPTXエクスポートはPhase 2送り可とする。

**ユーザー体験の意味**:

- 紹介者が本番後にPowerPointで自由にデザインを詰めて、友達へのプレゼントや記念物にできる
- 「AI生成だけど最終的に自分のものにできる」という所有感を提供

**NG内容のチェック**:

#### 7.3.4 AIプロンプト設計方針（Gemini 3 Flash）

**システムプロンプト**:

- 掛け合い誘発型の文体（紹介者セリフ案 + 被紹介者リアクション想定）を出力フォーマットとして強制
- NGカテゴリ（容姿・スペック）を明示的に禁止
- 紹介者の回答文脈を崩さず、テンプレに沿って整形する役割に限定
- 出力言語: 日本語のみ
- 一文の長さを口頭ピッチ向けに最適化（長すぎず、読み上げやすいリズム）

**API呼び出し設定**:

- モデル: `gemini-3-flash-preview`（Google AI Studio 経由）
- `responseMimeType`: `application/json`
- `responseSchema`: スライド配列の構造化スキーマを指定
- `thinkingLevel`: `minimal`（コスト最適化、整形タスクには深い推論不要）
- フォールバック: 生成失敗時はテンプレの素のプレースホルダを表示して手動編集に戻す

**思想的ガードレール**:

- 質問には**紹介者が自分の言葉で答える**。AIは整形役
- 「回答が空のままAIに全部書かせる」ことはUI上不可（必須入力を設ける）
- AIは画像を見ない設計（テキスト入出力のみ）→ 内面マッチング思想との整合を維持

#### 7.3.5 内面エピソード誘導の仕組み

トモコイの核は「顔を出さず、内面で魅力を伝える」こと。しかし紹介者が自由記述で書くと「優しい人です」「面白い人です」といった抽象形容詞に流れがちで、内面は伝わらない。**質問フォームUI側で具体エピソードを引き出す仕掛け**を組み込む。

**UI上の誘導機構**:

1. **質問ガイダンステキスト**
   - 各質問の下に「容姿・スペックではなく、エピソードを書いてください」というガイドを常時表示
   - 良い回答例・避けてほしい回答例を折りたたみ展開で併記
   - 例（Slide 2）:
     - ✅ 良い例: 「キャンプにハマってて、先月も1人で奥多摩に行ってた」
     - ❌ 避けてほしい: 「いろんなことに興味があります」

2. **回答の具体性チェック（クライアント側）**
   - 文字数下限（例: 20文字以上）
   - 抽象形容詞だけの回答を検出して「もう少し具体的に書けますか？」とヒント表示
   - 検出ワードリスト例: 「優しい」「面白い」「いい人」「素敵」などが単体で使われた場合

3. **エピソード誘導プロンプト**
   - 各質問に「この質問に答える時のコツ」を併記
   - Slide 3（意外な一面）: 「人に言わない秘密や、意外な趣味、周りが知らないエピソードは？」
   - Slide 5（心に残ったエピソード）: 「その友達のことを好きだと感じた、具体的な出来事を一つ」

4. **AI側のガイダンス（Gemini 3 Flash プロンプト）**
   - 紹介者の回答が抽象的な場合、AIは「容姿・スペック」の方向には絶対に向かわず、「具体エピソードが提示されていない」旨を示唆する文面として整形する（= 編集で具体化する動機を作る）

**なぜこの設計が必要か**:

- 紹介者にとって一番書きやすいのは「優しい」「面白い」という抽象形容詞。しかしそれでは顔出しなしで魅力を伝えるのは不可能
- 顔面至上主義からの脱却は、**抽象形容詞至上主義からの脱却**でもある
- UI レベルで具体エピソードを強制することで、初期ユーザーが「どう書けばいいかわからない」問題を解消
- 結果、被紹介者の人柄が立体的に伝わる → マッチング成立後の会話も「エピソードを知っている状態」から始まる

**デモデーでのアピール軸**:

- この仕組みは「顔を見せない代わりに、中身を深く見せる」という思想を、UIレベルで実装した証拠
- 既存マチアプとの差別化要素として審査員に説明可能

- 萌え系アニメキャラ風（目が大きく、髪色が派手など）

- **VRoid Hub の CC0 / 商用可素材**: 一般向けに作られたセミリアル系VRMが多数

#### 7.5.1 スタンプリアクション詳細仕様

**スタンプ種類**: 4種（🤝共感 / ✨素敵 / 😂面白い / 👏拍手）

**UI配置**:

- スライドの下に横一列でスタンプボタン配置
- タップすると画面下から上へアニメーションでスタンプが浮かぶエフェクト
- 配信アプリ（TikTokライブ・Instagram Live等）の演出を踏襲

**送信制限**:

- 回数上限なし、連打OK
- ただし**クールタイム**を設ける（0.3〜0.5秒程度、サーバ側で制御）
- 連打スパム防止と演出の見やすさを両立

**匿名性**:

- **完全匿名**。誰が押したかは他の参加者に一切表示されない
- スタンプの**累計数もユーザーには表示しない**（スタンプはエフェクトのみ、数値は見えない）
- 盛り上がりは画面を流れるスタンプの量で視覚的に感じる設計

**送信権限**:

| ロール                         | プレゼン中のスタンプ送信                   | プレゼン中の音声                 |
| ------------------------------ | ------------------------------------------ | -------------------------------- |
| 登壇中の紹介者                 | 送信不可                                   | **ON**（メインプレゼン）         |
| 登壇中の被紹介者               | 送信不可                                   | **ON**（合いの手・リアクション） |
| 他のプレゼン枠参加者（待機中） | 送信可                                     | OFF                              |
| オーディエンス枠               | 送信可（投票権なし、盛り上げ役として参加） | OFF                              |

**集計・アワード連動**:

- スタンプ送信イベントはDBに保存（送信者・発表ペアID・スタンプ種類・タイムスタンプ）
- プレゼン終了後、発表ペアごとに獲得総数を集計
- **スタンプ獲得総数最多のペアに「盛り上がったで賞」を自動表彰**（オーディエンス送信分もカウント対象）
- 表彰はイベント終了時にアプリ内画面表示 + 登録メール宛に通知

**技術実装**:

- 送信: クライアント → Supabase Realtime Broadcast（低レイテンシ、永続化なし）
- 並行して: DB INSERT（集計用、非同期で可）
- 受信: 全参加者のクライアントが購読、浮かぶエフェクトをレンダリング
- クールタイム: クライアント側UI制御 + サーバ側のレート制限（IP/user_id単位）

### 7.6 マッチングアルゴリズム

**入力**: 被紹介者リスト（性別属性）、各被紹介者の投票リスト、推薦フラグ（UI表示用）
※ 紹介者は計算対象外（卓不参加）

**定式化**: **制約付きk-partition問題**として扱う

- 全被紹介者をk個のテーブル（各3〜4人）に分割する
- ハード制約:
  - 各テーブル {3, 4} 名（被紹介者 N=5 のみ {5} を例外許可）
  - 全被紹介者が必ずどこかのテーブルに配属される
  - 登壇ペア分離は紹介者不参加により自動充足
- ソフト目的関数（辞書式優先）:
  1. 男女 2:2 卓数を最大化（4人卓のうち2:2になる卓数、`other`含む卓は混合扱い）
  2. 同性のみ卓を最小化（混合卓最大化）
  3. 相互投票 rank 合計を最大化（rank1=3, rank2=2, rank3=1 の重み）

**アルゴリズム**: **2-opt法**

1. 初期解: 制約を満たすランダム分割
2. 改善: 2テーブル間で参加者1名ずつ交換し、辞書式スコアが改善する限り繰り返し
3. 端数が出る場合は既存テーブルに3名テーブルとして再配置

**実装**: **TypeScript純関数として自前実装**（200〜400行規模）

- Python `matching` ライブラリは不使用（定式化をSR/SM/HR問題ではなく制約付きk-partitionに変えた時点で、ライブラリ依存のメリットがゼロ）
- Next.js **Server Action**（または API Route）から同期呼び出し
- HTTP境界なし、request/response がTypeScript型で固まる
- マッチング計算結果のDB書き込みも同一トランザクション内で完結

**処理時間目標**: 20人規模で3秒以内（Vercel Serverless Functionタイムアウト以内）

### 7.7 マッチ成立・事後

| 機能                     | 詳細                                                                           | 優先度  |
| ------------------------ | ------------------------------------------------------------------------------ | ------- |
| マッチ通知               | **登録メールアドレス宛に送信**「○○さんがあなたへのプレゼントを準備しました🎁」 | MVP     |
| 相互承諾UI               | 双方が「デートに行く」を承諾（アプリ内操作）                                   | MVP     |
| **1:1 チャット機能**     | **マッチ成立ペアのテキストチャット（Supabase Realtime Broadcast）**            | **MVP** |
| **顔写真の相互公開機能** | **双方の合意が揃った時点で、事前に非公開保存していた顔写真が相互に公開される** | **MVP** |
| 断りの設計               | 「今は受け取らない」をグレー・サブCTAで配置                                    | MVP     |
| 紹介者通知               | メール通知「プレゼンが○人に届きました」のみ（マッチ成否は伝えない）            | MVP     |

**通知方式について**: 本プロダクトはWebアプリのため、OSレベルのプッシュ通知は使用しない。イベント中の案内は**アプリ内画面表示**、イベント終了後の非同期通知は**登録メールアドレス宛の送信**で実装する。Phase 2以降でPWA化した際にWeb Push通知も検討。

#### 7.7.1 1:1 チャット機能

**目的**: マッチ成立後の会話の場を、トモコイのプラットフォーム内に閉じて提供。外部連絡先の交換前に、アプリ内で安心して会話を始められる環境を作る。

**仕様**:

- マッチ成立ペアのみが利用可能
- テキストメッセージのみ（画像・ファイル送信は MVP 非対応）
- リアルタイム配信: Supabase Realtime Broadcast
- メッセージ履歴: Supabase Postgres に保存、過去ログは遡って閲覧可能
- 通報機能: 各メッセージ単位で通報可能（運営が確認して対応）
- ブロック機能: 相手をブロックするとチャット打ち切り、以降の表示停止

**MVP 非対応（Phase 2送り）**:

- ビデオ通話・音声通話
- 画像・スタンプ・絵文字送信（絵文字は文字入力として扱う）
- 既読表示・タイピング表示
- メッセージ検索

#### 7.7.2 顔写真の相互公開機能（事前非公開保存 → 合意公開）

**目的**: 「顔を出さない」原則を守りながら、双方が「顔を知りたい」と望んだ場合には段階的に開示できる導線を用意。顔面至上主義に戻らず、合意に基づく開示というコントロール感を保つ。

**運用フロー**:

```
[登録時]
ユーザーが顔写真を1枚アップロード（任意、ただしマッチ後公開したい場合は必須）
  ↓
Supabase Storage に RLS（Row Level Security）で非公開保存
本人以外は一切アクセス不可（運営も含む）
  ↓
[マッチ成立後、チャット画面内]
「顔写真を見せ合う」ボタンを両者がタップ
片方だけの場合は相手に通知せず、両方タップで公開成立
  ↓
[公開成立時]
チャット画面内に双方の顔写真が表示される
どちらかが「公開解除」を押すといつでも再び非公開化
```

**設計判断の論拠**:

- **事前アップロード方式**を採用した理由: ユーザーが合意のタイミングで急いでスマホ内を探さなくていい、UXが滑らか
- **「片方タップ」を相手に通知しない設計**: 相手から「見たがっている」プレッシャーを感じさせない。双方が自発的に押した時だけ成立
- **公開解除が常に可能**: 「一度見せたら取り返しがつかない」恐怖を排除。トモコイ全体の「心理的安全性」の設計思想と一貫

**セキュリティ要件**:

- Supabase Storage の RLS ポリシー: 本人以外（運営含む）は一切アクセス不可
- 公開成立時のみ、マッチング相手の認証トークンでサイン付きURLを発行
- 公開解除で即座にサイン付きURL無効化
- 画像の外部転送・スクショ対策は MVP では未実装（利用規約で禁止明記のみ）

**UI 上のガイダンス**:

- 「顔写真を見せ合う」ボタンには「双方が押したら公開されます。いつでも取り消せます」と明記
- 「この写真は運営からも見えない形で保管されています」と安心材料を示す

#### 7.7.3 今後の拡張（Phase 2 候補）

- ビデオ通話・音声通話（合意制、段階的開放）
- 「友達の友達」表示（マッチ相手が自分の知人と繋がっているか可視化）
- チャット内での「質問カード」機能（会話のきっかけ提供）
- 紹介者への「その後どうなったか」フィードバックループ（プライバシー配慮の上で）

## 8. 技術スタック

### 8.1 基盤

| 領域           | 技術                                                          |
| -------------- | ------------------------------------------------------------- |
| フロントエンド | Next.js 16 (App Router) + TypeScript                          |
| UI             | shadcn/ui + Tailwind CSS                                      |
| バックエンド   | Supabase（PostgreSQL + Auth + Realtime + Storage）            |
| ホスティング   | **Vercel 一元化**（Cloud Run・Supabase Edge Function 不使用） |

**デプロイ方針**: 全機能をVercelに集約する。Cloud Run・Edge Functionの IAM・認証トークン・コールドスタートwarm-up運用を排除。ローカル開発は Docker 不要、`pnpm dev` のみで全機能動作。

### 8.2 リアルタイム・通信

| 領域         | 技術                                                |
| ------------ | --------------------------------------------------- |
| 状態同期     | Supabase Realtime（WebSocket）                      |
| 音声通話     | WebRTC（プレゼン中2者 + 交流タイム3〜4人、SFU不要） |
| スライド同期 | Supabase Realtime Broadcast                         |

### 8.3 アバター・ML

| 領域             | 技術                               |
| ---------------- | ---------------------------------- |
| 3Dレンダリング   | Three.js + React Three Fiber       |
| VRM読み込み      | @pixiv/three-vrm                   |
| 表情トラッキング | MediaPipe Face Landmarker (JS)     |
| 表情→VRM         | カスタムブレンドシェイプマッピング |

### 8.4 マッチング

| 領域                   | 技術                                                                                         |
| ---------------------- | -------------------------------------------------------------------------------------------- |
| マッチングアルゴリズム | **TypeScript純関数として自前実装**（制約付きk-partition問題を2-opt法で解く、200〜400行規模） |
| 呼び出し               | **Next.js Server Action**（または API Route）                                                |
| デプロイ               | Vercel一元化（Cloud Run・Supabase Edge Function 不使用）                                     |

**設計判断**:

1. **ライブラリ非依存**: SR/SM/HR問題ではなく制約付きk-partitionとして定式化した時点で、Python `matching` ライブラリに依存するメリットがゼロ。写経対象は自前の200〜400行
2. **デプロイ先が1つ減る**: Vercel単体に集約。Cloud RunのIAM・認証トークン・コールドスタートwarm-up運用がまるごと不要
3. **型が API境界を守る**: Server Action経由ならrequest/responseがTypeScript型で固まる。HTTP + JSONだと境界でバグる余地が残る
4. **担当の負担減**: 投票UI・マッチング・通知・推薦フラグを抱えているメンバーDから、インフラ設定1個減らせる
5. **Supabase Edge Functionも不要**: マッチング以外に使う予定がないので、Denoランタイムごと構成から外せる

**副次的にクリーンになる点**:

- Edge Function → Cloud Run → DB の三角形が、Server Action → DB の直線になる
- マッチング計算結果のDB書き込みも同一トランザクション内で完結できる
- ローカル開発でDocker不要、`pnpm dev` だけで全機能動く

### 8.5 AI / 生成系

| 領域             | 技術                                                                                   |
| ---------------- | -------------------------------------------------------------------------------------- |
| スライド文面生成 | **Google Gemini 3 Flash**（モデル名: `gemini-3-flash-preview`、Google AI Studio 経由） |
| SDK              | `@google/genai`（Google Gen AI SDK、Next.js Server Action から呼び出し）               |
| APIキー管理      | Google AI Studio で発行した APIキーを Vercel 環境変数 `GEMINI_API_KEY` に格納          |
| 用途             | スライド文面生成のみ（紹介者回答 → 掛け合い誘発型スライド文面への整形）                |
| 出力制御         | `responseMimeType: "application/json"` + `responseSchema` で構造化出力強制             |
| 推論設定         | `thinkingLevel: "minimal"`（整形タスクは深い推論不要、コスト・レイテンシ最適化）       |
| ガードレール     | システムプロンプトでNGカテゴリ（容姿・スペック）を禁止 + 日本語出力強制                |
| フォールバック   | 生成失敗時はテンプレの素のプレースホルダを表示して手動編集に戻す                       |

**Gemini 3 Flash 選定理由**:

- 無料tierあり（ハッカソン期間中の開発・デモが実質無料、Flash系は2026年4月以降も無料枠維持）
- 日本語品質が高い（Google系モデルの強み、日本人ユーザー限定のトモコイと相性良い）
- 知識カットオフが新しい（2025年1月）、現代の若者トレンドも反映しやすい
- 構造化JSON出力がネイティブサポート（テンプレ適合バリデーションをAPIレベルで担保）
- Vercel Server Action から `@google/genai` SDK でシンプルに呼び出せる（Vercel一元化の方針と整合）

**AI Studio vs Vertex AI**: トモコイではGoogle AI Studio（Gemini Developer API）を採用。Vertex AIは企業向けSLA・GDPR準拠・GCPエコシステム統合が必要な場合に使うが、MVPでは過剰。APIキー発行だけで即利用可能なAI Studioが適切。

**補足**: マッチングアルゴリズムは AI非依存（TS純関数 + Server Action）。AI使用はスライド文面生成に限定する。

### 8.6 ファイルストレージ・エクスポート・メッセージング

| 領域                                 | 技術                                                                                |
| ------------------------------------ | ----------------------------------------------------------------------------------- |
| スライド構造データ                   | Supabase Postgres（スライドJSONをDBに保存）                                         |
| スライド内画像アセット               | Supabase Storage（紹介者がアップロードした任意画像）                                |
| **プロフィール顔写真（非公開保存）** | **Supabase Storage + RLS**（本人以外アクセス不可、合意成立時のみサイン付きURL発行） |
| PPTXエクスポート生成                 | **pptxgenjs**（Node.jsネイティブ、Vercel Server Actionから直接呼び出し）            |
| PPTX日本語フォント                   | Noto Sans JP をpptxgenjs経由で埋め込み                                              |
| **マッチ後チャット**                 | **Supabase Realtime Broadcast（配信）+ Postgres（履歴保存）**                       |

**PPTXエクスポートの実装方針**:

- Server Action で DBから読み込んだスライド構造データ + 画像URL を元にPPTXを生成
- 生成したPPTXはレスポンスとして直接ダウンロード（一時ファイル保存不要）
- テンプレに応じた基本レイアウト（背景色・フォント・画像配置）を埋め込み
- MVP優先度はWeek 3後半のNice-to-have

**プロフィール顔写真の RLS 設計**:

- テーブル: `profile_photos`（user_id, storage_path, uploaded_at）
- RLS ポリシー:
  - SELECT: `user_id = auth.uid()` のみ許可（本人のみ閲覧）
  - 合意公開時は、サーバー側で**マッチング相手の user_id をチェック + 双方の合意フラグを確認**してから、期限付き署名URLを生成
- 公開解除時は、署名URLの TTL を短く設定しておくことで自然失効 + サーバー側で即座に無効化

**チャットのデータ設計**:

- テーブル: `matches`（マッチング結果）、`match_messages`（メッセージ）、`photo_reveal_consents`（顔写真公開の合意状態）
- Realtime Broadcast: マッチペアごとに独立したチャネル
- 過去ログ: Postgres に永続保存、DB制約でマッチング当事者のみ SELECT 可能

## 17. 信頼・安全設計

顔出しなしマッチングは構造的に「おじさん凸」「詐欺師流入」リスクを抱える。トモコイはこれを **マッチング後に顔画像を出すこと** で防ぐ。以下、段階的な施策として整理する。

### 17.1 設計思想

- 被紹介者は最終マッチングするまで**顔・名前を隠す**
- 技術認証は最小限に、**社会関係資本と文脈**で信頼を積み上げる
- マイナンバー連携は**不採用**（「容姿を隠す」と「国民番号で身元確認」は思想的に両立しにくく、ユーザー心理的なハードルも高すぎる）
- マッチング後にお互いに顔写真を見せ合うことで実際に合う前の不安を減らす

### 17.2 段階的施策

| フェーズ                    | 施策                                                                                                                                                                                   |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MVP（Hack-1提出時点）**   | ① 主催者による**完全招待制**<br>② ハッカソン主催という文脈そのものを**初期信頼スタンプ**として活用<br>③ 交流タイム中の**通報ボタン**                                                   |
| **Phase 1（実運用開始時）** | ① **大学メール / 企業メールドメイン認証**（主軸）<br>② 女性参加者向けセーフガード（運営ROM入室、24時間クーリングオフ、女性紹介者ペア推奨バッジ）<br>③ **参加履歴バッジ**による信頼累積 |
| **Phase 2（スケール期）**   | ① 自治体・大学・企業との**提携による団体認証**<br>② SNS連携による紹介者の身元強化<br>③ 運営観覧者制度の本格化                                                                          |

### 17.3 メールドメイン認証を主軸に据える理由

- マイナンバー連携より**ハードル1/10**（ユーザー登録障壁が低い）
- なりすましコストは十分に高い（大学発行メール・企業発行メールは転売・偽造困難）
- **初期コミュニティのターゲット層（大学生・若手社会人）と完全に一致**
- 思想的にクリーン（国民番号との紐付けなし）

### 17.4 審査員への説明（§11.3 軸5と連動）

> **「トモコイは顔を隠す。だからこそ、紹介者の身元は隠さない。**
> **被紹介者の匿名性と、紹介者の実名性。**
> **このアシンメトリが、友人紹介文化の再構築を実装レベルで可能にする。」**
